# DarkWG

VPN-туннель с обфускацией трафика (32-битные заголовки H1-H4, junk-пакеты
Jc/Jmin/Jmax, маскировка размеров S1/S2) + REST API и CLI для управления
пирами. Работает в собственных Docker-контейнерах (`darkwg`, `darkwg-nginx`),
полностью независимо от любой другой инфраструктуры на сервере — ничего из
существующих сервисов (например, RemnaWave) не трогается и не используется
совместно.

## Требования

- Ubuntu 24.04 (noble), root-доступ
- Docker (поставится автоматически, если его ещё нет)
- Открытый исходящий доступ в интернет на сервере для установки пакетов
- Один свободный UDP-порт для тоннеля (по умолчанию выбирается случайно)

## Установка

```bash
git clone https://github.com/thedarkkness/DarkWG.git
cd DarkWG
chmod +x install.sh
sudo ./install.sh
```

Скрипт сначала спросит **режим установки**:

1. **Всё на одном сервере** — туннель и API в одном месте, бот/панель тоже
   здесь же. API доступен только на `127.0.0.1:8765`, ничего лишнего не
   настраивается, контейнер `darkwg-nginx` не поднимается вообще.
2. **Эта нода отдельно от сервера управления** — бот/панель работают на
   другом сервере. Дополнительно спросит:
   - домен для API этой ноды
   - публичный IP сервера управления (доступ к API будет ограничен только им)
   - email для уведомлений Let's Encrypt (можно пропустить)

   Сертификат выпускается через **ACME HTTP-01 (standalone)**. Если порт 80
   свободен — просто и быстро. Если занят (например, твоим `remnawave-nginx`
   на `network_mode: host`) — скрипт сам найдёт этот контейнер, на секунды
   остановит его на момент выпуска и сразу поднимет обратно; `--pre-hook`/
   `--post-hook` сохраняются certbot'ом в конфиг продления, так что и при
   автообновлении через `certbot.timer` это будет происходить так же само,
   без участия. Поднимается отдельный контейнер `darkwg-nginx` на отдельном
   порту (по умолчанию 8443), доступ к нему ограничен через `allow`/`deny`
   только с IP сервера управления.

Публичный IP/домен сервера для самого тоннеля (то, к чему подключаются
клиенты) определяется автоматически и предлагается как вариант по умолчанию —
достаточно нажать Enter.

## Архитектура

```
              ┌─────────────────────────┐
              │     контейнер darkwg     │
много IP/UDP →│  awg-quick (туннель)     │
              │  + REST API (127.0.0.1)  │
              └─────────────┬───────────┘
                             │ 127.0.0.1:8765 (host network)
              ┌──────────────┴───────────┐
   режим 2 →  │   контейнер darkwg-nginx  │ → HTTPS наружу, только с IP сервера управления
              └───────────────────────────┘
```

Оба контейнера запускаются с `network_mode: host` — это обязательно для
`darkwg`, так как он создаёт и управляет сетевым интерфейсом ядра (`darkwg0`)
и правилами `iptables`/`sysctl` на уровне хоста; изолированный bridge-network
для этого не подходит (по той же причине `remnanode` у RemnaWave тоже без
явных портов в `docker ps` — тот же паттерн).

Сами бинарники `awg`/`awg-quick` не пересобираются и не ставятся повторно
внутри образа — монтируются с хоста как read-only тома (PPA Amnezia — это
Ubuntu-репозиторий, не совместим напрямую с debian-slim базой образа).
Kernel-модуль `amneziawg` в принципе не может жить "в контейнере" — это
свойство хост-ядра, ставится `install.sh` один раз на хосте.

## Управление пирами

**Через CLI (внутри контейнера):**

```bash
docker compose -f docker-compose.generated.yml exec darkwg \
  python3 scripts/darkwg_cli.py add-peer \
  --telegram-user-id 123456789 --ttl-days 30 --out /etc/darkwg/peers/user_123456789

docker compose -f docker-compose.generated.yml exec darkwg \
  python3 scripts/darkwg_cli.py list-peers

docker compose -f docker-compose.generated.yml exec darkwg \
  python3 scripts/darkwg_cli.py remove-peer --id 5
```

(Файлы конфигов/QR при этом появляются и на хосте в `/etc/darkwg/peers/` —
это общий volume.)

**Через API (для бота или своей панели):**

Все эндпоинты, кроме `/health`, требуют заголовок `X-API-Key` (значение — в
`/etc/darkwg/api.env` на сервере, в переменной `DARKWG_API_KEY`).

| Метод  | Путь                     | Что делает                                       |
|--------|--------------------------|---------------------------------------------------|
| POST   | `/peers`                 | Создать пира, вернуть `.conf` + QR в base64       |
| GET    | `/peers`                 | Список всех пиров                                 |
| GET    | `/peers/by-user/{tg_id}` | Пиры конкретного telegram-пользователя            |
| GET    | `/peers/{id}`            | Один пир + актуальный конфиг                       |
| DELETE | `/peers/{id}`            | Отозвать пира                                      |
| POST   | `/peers/{id}/disable`    | Временно выключить                                 |
| POST   | `/peers/{id}/enable`     | Включить обратно                                   |
| POST   | `/peers/{id}/extend`     | Продлить подписку на N дней (`{"add_days": 30}`)  |
| GET    | `/stats`                 | Живая статистика handshake/трафика                 |

Подписки с истёкшим `expires_at` отключаются автоматически — встроенная
фоновая проверка раз в 5 минут отзывает доступ у пиров, чей срок истёк.

Пример запроса:

```bash
curl -s -X POST http://127.0.0.1:8765/peers \
  -H "X-API-Key: <ключ из api.env>" \
  -H "Content-Type: application/json" \
  -d '{"telegram_user_id": 123456789, "ttl_days": 30}'
```

## Интеграция с ботом

Шаблон вызова API из aiogram-бота — см. `examples/bot_integration_example.py`.

- **Режим 1** (бот на этом же сервере) — используй `http://127.0.0.1:8765`
- **Режим 2** (бот на другом сервере) — используй `https://<домен_API>:<порт>`,
  именно этот URL `install.sh` выводит в конце установки

## Продление сертификата (режим 2)

Полностью автоматическое — `certbot.timer` сам проверяет дважды в день и
продлевает сертификат в течение 30 дней до истечения. Если порт 80 в момент
продления занят тем же контейнером, что и при первом выпуске — он будет так
же временно остановлен и поднят обратно (через те же `--pre-hook`/`--post-hook`,
сохранённые в `/etc/letsencrypt/renewal/<домен>.conf`). Дополнительных действий
не требуется.

## Хранение приватных ключей

API/CLI хранят приватный ключ клиента в SQLite (volume `darkwg-data`,
смонтирован в `/opt/darkwg/data` внутри контейнера), чтобы можно было
повторно выдать конфиг без отзыва старого.
