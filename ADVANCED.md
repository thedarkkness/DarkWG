# DarkWG — подробности

## Требования

- Ubuntu 24.04 (noble), root-доступ
- Docker (поставится автоматически, если его ещё нет)
- Открытый исходящий доступ в интернет на сервере для установки пакетов
- Один свободный UDP-порт для тоннеля (по умолчанию 443)

## Установка из исходников (без one-liner)

```bash
git clone https://github.com/thedarkkness/DarkWG.git
cd DarkWG
chmod +x install.sh
sudo ./install.sh
```

`install.sh`, запущенный через `bash <(curl ...)`, сам определяет, что рядом
нет остальных файлов репозитория, клонирует всё во временную папку и
перезапускается уже оттуда — оба способа равнозначны по результату.

## Режимы установки

**Режим 1 — всё на одном сервере.** Туннель и API в одном месте, бот/панель
тоже здесь же. API доступен только на `127.0.0.1:8765`, ничего лишнего не
настраивается, контейнер `darkwg-nginx` не поднимается вообще.

**Режим 2 — нода отдельно от сервера управления.** Бот/панель работают на
другом сервере. Дополнительно спрашивает:
- домен для API этой ноды
- публичный IP сервера управления (доступ к API будет ограничен только им)
- email для уведомлений Let's Encrypt (можно пропустить)

Сертификат выпускается через ACME HTTP-01 (standalone). Если порт 80
свободен — быстро и просто. Если занят (например, `remnawave-nginx` на
`network_mode: host`) — скрипт сам находит этот контейнер, на секунды
останавливает его на момент выпуска и сразу поднимает обратно;
`--pre-hook`/`--post-hook` сохраняются certbot'ом в конфиг продления, так что
и при автообновлении через `certbot.timer` это происходит так же само, без
участия. Поднимается отдельный контейнер `darkwg-nginx` на отдельном порту
(по умолчанию 8443), доступ ограничен через `allow`/`deny` только с IP
сервера управления. Продление полностью автоматическое — `certbot.timer`
проверяет дважды в день и продлевает в течение 30 дней до истечения,
дополнительных действий не требуется.

## Архитектура

```
              ┌─────────────────────────┐
              │     контейнер darkwg     │
много IP/UDP →│  awg-quick (туннель)     │
              │  + REST API (127.0.0.1)  │
              └─────────────┬───────────┘
                             │ 127.0.0.1:8765 (host network)
              ┌──────────────┴───────────┐
   режим 2 →  │   контейнер darkwg-nginx  │ → HTTPS наружу, только с IP сервера управления
              └───────────────────────────┘
```

Оба контейнера на `network_mode: host` — это обязательно для `darkwg`, так
как он создаёт и управляет сетевым интерфейсом ядра (`darkwg0`) и правилами
`iptables`/`sysctl` на уровне хоста; изолированный bridge-network для этого
не подходит (по той же причине `remnanode` у RemnaWave тоже без явных портов
в `docker ps` — тот же паттерн).

Бинарники `awg`/`awg-quick` не пересобираются внутри образа — монтируются с
хоста как read-only тома (PPA Amnezia — Ubuntu-репозиторий, не совместим с
debian-slim базой образа). Kernel-модуль `amneziawg` ставится `install.sh`
один раз на хосте — это свойство хост-ядра, не контейнера.

## Полный список API-эндпоинтов

Все, кроме `/health`, требуют заголовок `X-API-Key`.

| Метод  | Путь                     | Что делает                                       |
|--------|--------------------------|---------------------------------------------------|
| POST   | `/peers`                 | Создать пира, вернуть `.conf` + QR в base64       |
| GET    | `/peers`                 | Список всех пиров                                 |
| GET    | `/peers/by-user/{tg_id}` | Пиры конкретного telegram-пользователя            |
| GET    | `/peers/{id}`            | Один пир + актуальный конфиг                       |
| DELETE | `/peers/{id}`            | Отозвать пира                                      |
| POST   | `/peers/{id}/disable`    | Временно выключить                                 |
| POST   | `/peers/{id}/enable`     | Включить обратно                                   |
| POST   | `/peers/{id}/extend`     | Продлить подписку на N дней (`{"add_days": 30}`)  |
| GET    | `/stats`                 | Живая статистика handshake/трафика                 |

Подписки с истёкшим `expires_at` отключаются автоматически — фоновая
проверка раз в 5 минут отзывает доступ у пиров, чей срок истёк.

## Интеграция с ботом

Шаблон — `examples/bot_integration_example.py`.

- Режим 1 (бот на этом же сервере): `http://127.0.0.1:8765`
- Режим 2 (бот на другом сервере): `https://<домен_API>:<порт>` —
  именно этот URL выводит `install.sh` в конце установки

## Хранение приватных ключей

API/CLI хранят приватный ключ клиента в SQLite (volume `darkwg-data`,
`/opt/darkwg/data` внутри контейнера), чтобы можно было повторно выдать
конфиг без отзыва старого.

## Переустановка на сервере, где уже что-то стоит

`install.sh` при повторном запуске **перегенерирует ключи сервера и
параметры обфускации заново** — все уже выданные клиентские конфиги
становятся недействительными. Если на сервере уже поднят `darkwg0` не через
Docker (например, вручную через `darkwg-quick up`) — опусти его перед
запуском, иначе контейнер не применит новый конфиг (поднимает интерфейс
только если его нет):

```bash
darkwg-quick down /etc/darkwg/darkwg0.conf 2>/dev/null || true
sudo ./install.sh
```
